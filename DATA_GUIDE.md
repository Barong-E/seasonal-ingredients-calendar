# 📚 제철 식재료 및 레시피 데이터 관리 가이드

이 문서는 '띵동 제철음식' 앱의 식재료와 레시피 데이터를 추가하거나 수정할 때 반드시 지켜야 할 규칙들을 정리한 **데이터 관리 비법서**입니다. 나중에 새로운 식재료를 추가할 때 이 문서를 참고하면 실수를 줄일 수 있습니다.

> [!IMPORTANT]
> **🇰🇷 대한민국 제철 표준 준수**: 이 앱의 모든 식재료 제철 정보는 **대한민국의 기후와 제철 시기**를 표준으로 합니다. 새로운 데이터를 추가할 때도 한국의 산지 출하 시기를 기준으로 작성해 주세요.
> 
> **🥘 식재료-레시피 필수 연동**: 식재료 데이터를 추가하거나 업데이트할 때는 반드시 해당 식재료를 활용한 **실제 조리 과정이 있는 레시피를 최소 1개 이상** 세트로 추가해야 합니다. 특히 **`popular_dish`(대표 요리)에 기재된 모든 요리는 반드시 `recipes.json`에 레시피가 존재하고 `recipe-mapper.js`에 매핑되어야 합니다.**
> 
> **🚫 레시피 없는 요리명 기재 금지**: 대표 요리에 이름을 올렸다는 것은 앱이 사용자에게 해당 요리법을 제공하겠다는 약속입니다. 레시피를 제공할 수 없는 요리(예: 단순 섭취 방식, 외부 링크 없는 요리 등)는 절대 대표 요리 목록에 넣지 마세요.

## 0. 🔍 식재료 선정 기준 (Selection Criteria)
신규 식재료를 추가할 때는 아래 기준을 우선순위로 하여 선정합니다.

1.  **중복 등록 방지 (Check Duplicates) [엄격 준수]**: 
    - **[AI 필수 절차]**: 후보를 선정하기 전에 반드시 `grep_search` 또는 `view_file`을 통해 `public/data/ingredients.json`에서 해당 이름(`name_ko`)이 존재하는지 1차 확인합니다.
    - **[더블 체크]**: 검색 결과가 없더라도, 비슷한 이름(예: '두릅' vs '참두릅')이나 카테고리가 겹치는지 파일의 전체 목록을 훑어보며 재확인해야 합니다. 
    - 이미 등록된 식재료를 사용자에게 추천 후보로 제안하는 것은 데이터 무결성을 해치는 중대한 실수로 간주합니다.
2.  **계절적 균형 유지 (Seasonal Balance) [중요]**: 
    - 특정 월(특히 현재 시점의 월)에만 식재료가 쏠리지 않도록 주의합니다.
    - **[AI 필수 절차]**: 식재료 제안 전, `ingredients.json`의 전체 월별 데이터 분포를 파악합니다. 상대적으로 데이터가 부족한 계절(봄, 여름, 가을, 겨울)이나 특정 월의 식재료를 우선적으로 발굴하여 전체적인 균형을 맞춥니다.
3.  **대중적 소비량 고려 (Popularity)**: 대한민국 국민들이 일상적으로 가장 많이 소비하고 친숙하게 느끼는 식재료를 우선적으로 등록합니다.
4.  **최신 트렌드 반영 (Trending)**: 최근 방송이나 SNS에서 화제가 되고 있는 식재료를 적극 발굴합니다.
5.  **사용자 승인 절차 (User Approval)**: 식재료 후보를 선정하면 검색 결과(중복 없음 확인 보고)와 선정 이유(계절적 균형 포함)를 함께 설명하고 승인을 받습니다.

## 1. 📂 파일 위치 및 역할
- `public/data/ingredients.json`: 식재료의 기본 정보(이름, 효능, 보관법 등) 저장
- `public/data/recipes.json`: 요리법 상세 정보(재료, 조리 단계 등) 저장
- `assets/recipe-mapper.js`: 식재료 상세 페이지와 레시피 페이지를 연결해 주는 '단어장'
- `public/images/`: 식재료 및 레시피 프로필 이미지 저장소

## 2. 📝 데이터 규격 (JSON Schema)

### 🥬 식재료 (`ingredients.json`)
반드시 아래와 같은 이름표(Key)를 사용해야 합니다.
```json
{
  "name_ko": "식재료 이름(한글)",
  "category": "해산물/채소/과일/기타",
  "image": "image_name.png",
  "description_ko": "식재료 설명",
  "preparation_ko": "손질법",
  "storage_refrigerator": "냉장보관법",
  "months": [1, 2, 12], // 제철 달(월) 정보 (숫자 배열)
  "popular_dish": "대표 요리 이름 (쉼표로 구분)"
}
```

### 🍲 레시피 (`recipes.json`)
**[🔥 극히 주의]** 식재료 데이터(`ingredients.json`)와 혼동하여 레시피 데이터를 잘못된 규격으로 작성하는 치명적 실수가 매우 잦습니다! 앱 화면이 완전히 망가질 수 있으니 아래 4가지를 무조건 지켜주세요.
1. **이름**: `name_ko`가 아니라 무조건 **`name`**을 사용합니다. (재료 목록 안의 개별 재료명도 `name` 사용)
2. **시간**: `time`이 아니라 무조건 **`cookTime`**을 사용하며, 반드시 '분'을 포함한 문자열(예: "20분")로 적어야 합니다.
3. **순서**: `steps`는 단순 문자열 배열(`["1단계", "2단계"]`)이 아니라, 반드시 **객체 배열(`[{"step": 1, "description": "1단계 내용"}, ...]`)** 형태여야 합니다.
4. **구분**: 주재료와 양념을 반드시 분리하여 각각 `ingredients`와 `seasoning` 필드에 나누어 작성해야 합니다.

```json
{
  "id": "recipe-unique-id", // 영문 소문자와 하이픈(-)만 사용
  "name": "요리 이름", // 절대 name_ko를 쓰지 마세요!
  "category": "해산물 등",
  "description": "요리 요약 설명",
  "servings": 2, // 숫자만
  "cookTime": "20분", // 반드시 '분' 포함 문자열
  "difficulty": "하/중/상",
  "ingredients": [ // 주재료 (고기, 생선, 채소, 물 등)
    { "name": "재료명", "amount": "양(예: 100g)" }
  ],
  "seasoning": [ // 양념 (간장, 고춧가루, 설탕, 마늘, 참기름 등)
    { "name": "양념명", "amount": "양(예: 1큰술)" }
  ],
  "steps": [
    { "step": 1, "description": "첫 번째 단계 설명" },
    { "step": 2, "description": "두 번째 단계 설명" }
  ]
}
```

## 3. 🖼️ 이미지 생성 가이드 (DALL-E 프롬프트)
식재료 이미지를 생성할 때는 기존 앱의 **만화/일러스트 톤앤매너**를 유지해야 합니다.
- **추천 프롬프트 스타일**: 
  > `Cute 2D cartoon style vector illustration of [식재료 영문명], bright colors, clean white background, simple outlines, food icon style`
- **⚠️ 중요 (여백 조절)**: 이미지가 너무 작게 보이지 않도록 **피사체가 화면에 꽉 차게(Filling the frame)** 생성해야 합니다. 프롬프트에 `close up`, `filling the frame with minimal margins`와 같은 키워드를 추가하여 여백이 너무 많이 남지 않도록 조절하세요.

## 4. 🔗 레시피 무결성과 매핑 규칙 (`recipe-mapper.js` 및 `ingredients.json`)
새로운 식재료와 레시피를 추가할 때 가장 많이 발생하는 "고립된 레시피(미아)"나 "연결 불일치" 오류를 막기 위한 핵심 규칙입니다.

1. **이름의 절대적 일치**: `ingredients.json`의 `popular_dish`에 적힌 요리 이름과 `recipes.json`의 `name`은 띄어쓰기까지 **정확하게 100% 일치**해야 합니다. (예: 추천 요리에 '토마토 스파게티'를 적어두고 실제 레시피 이름을 '토마토 마리네이드'로 적으면 안 됩니다.)
2. **누락 없는 1:1 매핑**: `popular_dish`에 나열된 모든 요리는 `recipe-mapper.js`에 키(Key) 값으로 등록되어야 하며, 해당 키가 가리키는 ID는 반드시 `recipes.json`에 존재해야 합니다.
3. **고립 방지**: `recipes.json`에 새롭게 추가된 레시피는 명절용 특수 음식(`holidays.json` 연동)이 아닌 이상, 최소 1개 이상의 식재료(`ingredients.json`)의 `popular_dish`에서 자신을 가리키도록 해야 합니다. 아무도 추천하지 않는 레시피는 만들지 마세요.

```javascript
const mapping = {
  '정확하게 일치하는 요리 이름': 'recipe-unique-id',
  ...
};
```

## 5. 🔄 반영 및 배포 절차 (필수!)
데이터를 수정한 후에는 다음 3단계를 **반드시 순서대로** 실행해야 합니다.

1.  **캐시 버전 올리기**: `assets/script.js`, `assets/ingredient.js`, `assets/recipe.js` 내의 `v14` 같은 버전 숫자를 한 단계 높입니다 (예: `v15`).
2.  **서비스 워커(PWA 캐시) 갱신**: (매우 중요) `recipe-mapper.js` 등 앱의 주요 화면 로직이나 자바스크립트를 수정했다면, 반드시 최상단 폴더의 `service-worker.js`에서 `const VERSION = 'vX';`의 **버전을 한 단계 올려야 합니다.** 그래야 상용 앱에서 기존에 저장된 캐시(단어장 포함)가 초기화됩니다.
3.  **안드로이드 동기화**: 터미널에서 `npm run build && npx cap sync`를 실행합니다.
4.  **저장소 업데이트**: `git add . && git commit -m "...내용..." && git push`를 실행하여 서버에 올립니다.

## 💡 자주 하는 실수 (Pitfalls)
- ❌ `cookTime`에 '20'만 쓰고 '분'을 안 쓰는 경우 (화면 출력 오류)
- ❌ 레시피의 `name`을 `name_ko`로 잘못 적는 경우 (데이터 로드 실패)
- ❌ `steps`를 그냥 글자 배열(`["1단계", "2단계"]`)로만 적는 경우 (기존 앱은 `{step: 1, description: ""}` 형태를 선호)
- ❌ 중복된 식재료(예: 두릅 vs 참두릅)를 추가하는 경우
- ❌ **[주의] 칼로리 및 보관법 누락**: 식재료 추가 시 검색을 통해 칼로리 정보를 반드시 기입하고, 냉장뿐만 아니라 **냉동 보관 가능 여부**를 전수 조사하여 누락 없이 작성해야 합니다.
- ❌ **[주의] 주재료/양념 미분리**: 양념 재료(간장, 마늘 등)를 `ingredients`에 섞어 넣지 마세요. 화면에 깔끔하게 구분되어 표시되도록 반드시 `seasoning` 필드를 사용해야 합니다.
- ❌ **[주의] 레시피 연동 미흡**: `popular_dish`에 나열된 모든 음식은 `recipe-mapper.js`에 등록되어야 하며, 연결되지 않는 요리가 없도록 `recipes.json`을 보강해야 합니다.

## ✅ 데이터 등록 전 최종 체크리스트 (Must-Check List)
배포 가이드(5단계)를 실행하기 전, 아래 항목을 하나하나 체크하며 데이터의 완성도를 검증하세요.

1.  **[ ] 영양 정보**: `calories_per_100g`과 `calories_per_serving`이 모두 입력되었는가?
2.  **[ ] 보관 3종 세트**: 실온(`storage_room_temp`), 냉장(`storage_refrigerator`), 냉동(`storage_freezer`) 보관법을 모두 조사하여 기입했는가? (냉동이 불가능하면 필드를 비워둡니다)
3.  **[ ] 레시피 100% 연동**: `popular_dish`에 적은 모든 요리가 `recipe-mapper.js`에 포함되어 있고, 실제 레시피 페이지로 연결되는가?
4.  **[ ] 이미지 스타일**: 배경이 하얀색이며, 피사체가 화면에 꽉 차게 생성되었는가?
5.  **[ ] 캐시 강제 갱신**: `script.js`의 `CACHE_KEY`와 모든 fetch URL의 버전 숫자를 기분 좋게(?) 한 단계 올렸는가?
6.  **[ ] 쿠팡 파트너스 연동**: 배포가 완료된 후, 사용자가 직접 생성한 쿠팡 파트너스 구매 링크(URL)를 적절한 위치에 추가하도록 안내를 받았는가?
